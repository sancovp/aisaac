cave-unicorn — How It Works (Deep Dive)

Every mechanism in the distribution pipeline — the nightly organs, the framework-first gate, the state machine on the graph — grounded in the source.

The story of why this exists: https://sancovp.github.io/aisaac/blog/cave-unicorn-the-story.html

This article explains what the cave-unicorn module actually does, mechanically, when you run it. Everything here comes from the module's own source code — the functions, the contracts, the CLI commands, and the state machine the system drives. If you want to evaluate whether to build or adapt this for your own system, this is the reference.

The architecture

The cave-unicorn module sits at the end of a content pipeline. Its job is to take a finished blog post and push it: onto the live website, into a social media pack, and then rendered as images. It does this through three independent CronAutomation organs that fire on a nightly schedule.

The module is structured as a Python package called cave_unicorn (pip-installable). Its internal components:

File Role
config.py Reads and writes the unicorn_config.json file — the persistent settings the module publishes against (site root path, git remote, default tags, etc.)
site.py The core render-and-publish logic: converts blog markdown to a full HTML post, inserts an index card, commits to git, optionally pushes
publish.py fire_site_publish — the CronAutomation code pointer. Two modes: direct (one file) or scan (CartON Blog_Request nodes done but not site-published)
socials.py fire_socials_pack — second CronAutomation. Reads site-published blogs, runs a heaven/minimax agent to write Twitter/LinkedIn/Discord posts and image prompts, renders the suite deterministically
images.py fire_socials_images — third CronAutomation. Runs a claude -p excalidraw agent to render hero, social-share, and thumbnail PNGs from the socials pack
announce.py Posts a Discord announcement after a pushed publish, via the cave_discord library (import-if-present)
dispatch.py Dispatches the heaven/minimax agent used by the socials pack leg
journey_suite.py Deterministic render engine: takes a filled journey_core.json + journey_blog.json and emits blog-aida.md + pack.md
cave_seam.py CAVE integration layer — emits or registers the CronAutomation schema. Never imports cave at module level
cli.py The v1 setup interface: init, show, set, publish, pending, publish-pending, enable-automation, and a few more

The CAVE layer law (stated in cave_seam.py): cave_unicorn is L9 (a library). CAVE (L7) consumes it, never the reverse. The module never imports cave at module level; it registers via register_with_cave(cave_agent) or emit_schema() to write the automation JSON into the hot-reload directory.

How it runs

The system runs as three CronAutomations on a nightly schedule:

02:00 — blog organ writes Blog_Request nodes with status=done
             (blog markdown at output_path)
             ↓
    02:37 — cave_unicorn.publish.fire_site_publish (scan mode)
             Blog_Request nodes {status: done, no site_published}
             → render markdown → write {blog_dir}/{slug}.html
             → insert card into {blog_dir}/index.html
             → git add/commit → (push=true) git pull --rebase --autostash + push
             → flip site_published + site_url + site_published_at on the node
             → announce to Discord
             ↓
    03:07 — cave_unicorn.socials.fire_socials_pack (scan mode)
             site_published nodes with no socials_packed
             → framework-first gate: skip if sibling journey_core.json
               lacks deep_dive_url and plugin_url (mark framework_missing, retry later)
             → dispatch heaven/minimax agent: fills journey_core.json +
               journey_blog.json + image-prompts.md
             → enforce canonical journey_core.json (fill-once: module overwrites
               any agent-written copy with the canonical version)
             → render_suite: deterministic blog-aida.md + pack.md
             → flip socials_packed + socials_pack_path on the node
             ↓
    03:37 — cave_unicorn.images.fire_socials_images (scan mode)
             socials_packed nodes with no images_rendered
             → run claude -p agent interpreting pack image-prompts.md as excalidraw
               diagrams, render with the Playwright script, verify PNGs exist
             → flip images_rendered + images_dir on the node
             ↓
    Ready-to-post queue: {socials_queue}/{slug}/pack.md + images/

Every automation has the never-raises contract: failures are captured in the return dict and on the CartON node, never thrown. This is because they run on the Heart tick — a crash would silently stop the pipeline.

Each automation also runs in direct mode: pass md_path or pack_path as a kwarg and it processes one file without touching CartON. This is the CLI path and the escape hatch for manual intervention.

Note on paths: Container-internal absolute paths in the source (e.g., default config values, environment variable defaults) are generalized to logical module paths in this article. The actual defaults reference container-internal directories.

The unicorn_config.json file (created by cave-unicorn init) holds the target site settings. Its keys: site_root, blog_dir, site_base_url, remote, branch, push, default_tags, discord_announce, discord_channel.

The state machine

Each blog post is tracked as a CartON Wiki node (a Blog_Request concept). The system drives it through a series of states by setting properties on the node:

status: done               (set by the blog organ — the input condition)
        ↓ [fire_site_publish scan]
    site_published: true       ← blog is on the live site
    site_url: <URL>           ← the live post URL
    site_published_at: <ISO>   ← timestamp
        ↓ [fire_socials_pack scan — gated]
    framework_missing: true     ← set when sibling journey_core lacks deep_dive_url/plugin_url
                                 (blocks dispatch; cleared when the links are wired in later)
        ↓ [pack_one succeeds]
    socials_packed: true
    socials_pack_path: <path>   ← path to pack.md in the socials queue
    socials_packed_at: <ISO>
        ↓ [fire_socials_images scan]
    images_rendered: true
    images_dir: <path>         ← path to the images/ directory (hero.png, social-share.png, thumbnail.png)
    images_rendered_at: <ISO>

Error states are written to the same nodes rather than raised: - site_publish_error — written by fire_site_publish when a publish fails - socials_pack_error — written by fire_socials_pack when a pack fails - images_error — written by fire_socials_images when rendering fails

The three automation scanners skip nodes that already have their output property set. A failed node will be retried on the next nightly scan once the error is resolved.

The invariants

These are the hard rules the module enforces in code — not suggestions.

Never-raises contract. Every fire_* automation catches exceptions and writes error strings to the CartON node's error property. A failure never crashes the Heart tick or blocks the rest of the queue.

Fail-loud git. publish_file in site.py runs git commands via _git(), which raises RuntimeError loudly if any git command fails. There are no silent fallbacks; a failed add/commit/pull/push surfaces immediately and is written to site_publish_error on the node.

Framework-first gate in socials. fire_socials_pack reads the sibling journey_core.json file next to the blog markdown. If neither deep_dive_url nor plugin_url is present, the node is marked framework_missing: true and skipped. This prevents packing a blog whose CTA/demo links point nowhere. The gate is cleared automatically on a subsequent scan once those fields are populated by the chapter pipeline.

Canonical core enforced on disk. When a canonical journey_core.json already exists on disk (from a prior fill), the socials pack's _enforce_canonical_core function overwrites whatever the heaven/minimax agent wrote with the canonical content before the suite renderer runs. The agent's output is used for its self-correction loop only; the module is the authority on the canonical JSON. This prevents a repeated bug where the agent retyped the core from memory and corrupted nine fields.

Blog markdown contract. site.py's split_front function parses the blog markdown with two hard rules: the first # heading line is the title, and the first non-empty block wrapped in *...* or **...** is the hook/subtitle. Everything else is the body. A markdown file without a # title heading raises ValueError immediately — it cannot silently publish as untitled.

Idempotent index card insertion. insert_card in site.py handles re-publish semantics: if a card with the same href already exists, it replaces the old card in place (status "updated") without bumping the post count. An identical card returns unchanged ("already_indexed"). A newly inserted card goes to the top of the card grid and the count is incremented.

Layer law. cave_unicorn never imports cave at module level. The CAVE seam is accessed through cave_seam.register_with_cave(cave_agent) (live agent path) or cave_seam.emit_schema() (writes automation JSON to the hot-reload directory for the Heart tick to pick up). Both paths are non-blocking — if cave is absent, register_with_cave returns False and emit_schema writes the JSON file directly.

Import-if-present throughout. cave_discord (for Discord announces), carton_mcp (for CartON access), and heaven_framework (for the heaven/minimax dispatch) are all imported lazily inside functions. If any is absent, the module logs the error and continues with a degraded but non-crashing result.
