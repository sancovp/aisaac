cave-unicorn — How It Works (Deep Dive)

The pipe from content the system produces onto the live GitHub Pages site — three organs, nightly, exactly how each part works.

The story of why this exists: cave-unicorn — the story

cave-unicorn is the UNICORN distribution module of the SANCREV SDK — the pipe from content the system produces onto the live GitHub Pages site, run as CAVE CronAutomations. This article explains exactly how each part works, drawn from the module docstrings and code.

The architecture

cave-unicorn has three organs, wired in sequence:

organ code_pointer nightly what it does
site publish cave_unicorn.publish.fire_site_publish 02:37 Blog_Request done → live post + index card + push + Discord announce
socials pack cave_unicorn.socials.fire_socials_pack 03:07 live blog → twitter/linkedin/discord posts + image prompts (heaven/minimax agent)
images cave_unicorn.images.fire_socials_images 03:37 pack → hero/social-share/thumbnail PNGs (claude -p excalidraw agent, visually verified)

(Table from: README.md — "The three organs (all E2E-proven 2026-07-10)" section)

The module is organized into these packages:

  • site.py — renders blog markdown into the live site's HTML post skeleton and inserts a card into the blog index.
  • publish.py — the fire_site_publish CronAutomation entrypoint: scans CartON for done-but-unpublished Blog_Request nodes and publishes them.
  • socials.py — the fire_socials_pack CronAutomation entrypoint: scans for published blogs without a socials pack, dispatches the blog-writing agent, then renders the suite deterministically module-side.
  • journey_suite.py — the deterministic render engine: takes a filled journey_core.json + journey_blog.json and produces blog-aida.md, pack.md, and the validated journey_core.json.
  • cave_seam.py — registers the three CronAutomations with CAVE via the import-if-present seam, and emits their schemas to the hot-reload dir.
  • images.py — fires the excalidraw image renderer.
  • announce.py — sends Discord announcements.
  • config.py / cli.py / dispatch.py — config loading, CLI interface, and agent dispatch.

How it runs

The nightly pipeline

Blog_Request {status: done, output_path: X.md, no site_published}
        → fire_site_publish (CronAutomation, nightly 02:37, after the 02:00 blog organ)
        → render X.md into the site's post skeleton  → {blog_dir}/{slug}.html
        → insert the blog-card into {blog_dir}/index.html (idempotent, count bumped)
        → git add/commit (+ pull --rebase --autostash + push when push=true)
        → flip the node: {site_published: true, site_url, site_published_at}

(Flow diagram from: README.md — "The flow" section)

Site publish (publish.py)

fire_site_publish has two modes:

Direct mode — called with md_path=...: publishes one markdown file immediately, renders it, writes the HTML post, inserts the index card, runs git, and (optionally) pushes. Returns a status dict including live_url.

Scan mode — called with no kwargs: queries CartON for Blog_Request-shaped nodes where status = 'done', an output markdown path exists, but site_published is null. For each match, it calls publish_file, then flips site_published: true, site_url, and site_published_at onto the node via CartON set_concept_properties.

publish_file (in site.py) performs these steps:

  1. Read the markdown. Apply link_map replacements (plain string substitution applied before render — used to point chapter-bundle sibling references at the published post URL).
  2. split_front — extract the # title line and the first *...* wrapped paragraph as the hook; everything else is the body.
  3. render_post — markdown-to-HTML via Python markdown with fenced-code and tables extensions, wrapped in the POST_TEMPLATE HTML shell (site-header nav, article, blog-CTA block, footer).
  4. Write {blog_dir}/{slug}.html.
  5. insert_card — insert (or replace) a blog-card in the index's <div id="posts"> grid. Replacement happens if a card with the same href already exists (so title/hook/tag edits propagate). Bumps the result-count ("N posts").
  6. Git: add the post + index, commit, and (if push=true) pull --rebase --autostash + push.

Any git failure raises loudly — no silent fallbacks.

Socials pack (socials.py)

fire_socials_pack also has two modes:

Direct mode — called with md_path=...: dispatches the blog-writing agent for that one file without touching CartON. Deliberately bypasses the framework gate — it is the operator's explicit hand.

Scan mode — queries CartON for nodes where site_published = true and socials_packed IS NULL. For each node:

  1. Framework-first gate (_framework_ready): reads the sibling journey_core.json (persisted next to the blog's output markdown) and checks that both deep_dive_url and plugin_url are non-empty. A node missing either is marked framework_missing and skipped — it will be retried on the next nightly scan once the chapter pipeline wires those fields.
  2. Agent dispatch (pack_one): builds the blog-writing prompt from DEFAULT_PROMPT_PATH (the canonical blog-organ prompt skill) and dispatches it via run_agent (BashTool, heaven/minimax model). The agent fills journey_core.json and journey_blog.json (and optionally image-prompts.md) in the per-slug output directory.
  3. Fill verification: checks both JSON files exist. If missing, returns failed.
  4. Canonical core enforcement (_enforce_canonical_core): overwrites whatever the agent wrote in journey_core.json with the byte-contents of the canonical file (the sibling core from the blog-organ run). This prevents the release-scan agent from re-typing the core from memory and corrupting fields like plah or IKZ. The only sanctioned addition is blog_url = live_url when the canonical left it unset.
  5. Suite render (journey_suite.render_suite): deterministic — takes the filled JSONs, produces blog-aida.md and pack.md. The agent's own render run is only its self-correction loop; the module is the authority.
  6. Flips socials_packed: true + socials_pack_path on the node (or socials_pack_error on failure).

Journey suite (journey_suite.py) — the deterministic render

The fill-once doctrine made real: the agent fills two JSONs; everything else renders deterministically in this module.

render_suite takes: - core_json_path — the validated journey_core.json - blog_json_path — the filled journey_blog.json - out_dir — where to write the artifacts - image_prompts_path — optional path to the agent's image-prompt fills

It loads the live models from the cave-discord-fork (integration/cave-discord-fork), not the stale absorbed copies in cave-discord itself. An origin check in _load_fork_modules refuses to render if the models resolve to a different path.

Output artifacts:

  • blog-aida.md — hero's-journey marketing render of JourneyBlog. Each model section (the slots named The Hook / Let Me Show You / Take Action...) renders as a four-part AIDA cycle (attention / interest / desire / action) with a bolded attention lead — the slot names themselves are NEVER published as headings (structure stays invisible). The piece closes with the CTA cycle as flowing copy and ONE funnel link — falling back to deep_dive_url then plugin_url if no funnel page exists yet.
  • pack.md — TwitterPostSet + LinkedInPost + DiscordJourneyPost renders from JourneyCore, plus the image prompts passthrough, plus a source footer.
  • journey_core.json — the re-validated core (persisted so downstream organs reuse it).

python3 -m cave_unicorn.journey_suite --contract prints the live field contract (name + REQUIRED/optional label per field for each model) — the agent prompt directs the agent to read this so the instructions can never drift from the models.

The state machine

A Blog_Request node (a CartON :Wiki node) progresses through these states:

(status: done, site_published: null)
        → site publish automation (02:37)
        → (status: published|staged, site_published: true, site_url: <url>, site_published_at: <ts>)
            → socials pack automation (03:07) — FRAMEWORK GATE
            → if deep_dive_url + plugin_url present on sibling core:
            → (socials_packed: true, socials_pack_path: <path>, socials_packed_at: <ts>)
            → images automation (03:37)
            → (images_rendered: true, ...)
            → OR: if sibling core missing deep_dive_url:
            → (framework_missing: true) — retried next night after chapter step wires it

Re-publish semantics: insert_card in site.py replaces an existing card for the same slug in-place (count untouched), so title/hook/tag edits propagate without bumping the post count.

Framework gate: a node without deep_dive_url on its sibling core is marked framework_missing and held — not packed — until the chapter pipeline wires the field. This prevents socials being sent for blogs whose CTA sections have no real links to point at.

The invariants

cave-unicorn holds these rules in code (not prose):

1. Blog markdown contract (site.py, split_front): - First # line = title (required; raises ValueError if absent). - First *...* wrapped paragraph after the title = hook (optional). - Everything else = body.

2. Fill-once, derive everywhere (journey_suite.py): - The agent fills only journey_core.json + journey_blog.json. - All renders are deterministic module-side — the agent's own render is only self-correction; the module is the authority. - python3 -m cave_unicorn.journey_suite --contract is the single source of truth for field requirements — the agent reads it to stay in sync.

3. Canonical core enforcement (socials.py, _enforce_canonical_core): - When a canonical journey_core.json exists (the sibling file from the blog-organ run), the module overwrites the agent's copy with the canonical bytes before the deterministic render. The only sanctioned field addition is blog_url = live_url when the canonical left it unset.

4. Framework-first gate (socials.py, _framework_ready): - A blog is not queued for socials until its sibling journey_core.json contains both deep_dive_url and plugin_url. Without them, the node is marked framework_missing and retried on the next scan.

5. Never-raises contract (all fire_* functions in publish.py and socials.py): - Failures are captured in the return dict (and on the CartON node where applicable), never thrown. This runs on the Heart tick — a crash would kill the nightly automation loop.

6. Import-if-present seam (cave_seam.py): - cave-unicorn never imports cave at module level. CAVE consumes unicorn; unicorn does not know about CAVE. Registration happens either via register_with_cave(live_agent) (live registration) or emit_schema() (schema emitted to the hot-reload dir for the Heart tick to pick up).

7. Git never falls back silently (site.py, _git): - Any non-zero return code from any git command raises RuntimeError loudly. No silent failures; push is optional but never half-done.

Source: all mechanics drawn from the module docstrings and code in application/cave-unicorn/cave_unicorn/ — specifically site.py, publish.py, socials.py, journey_suite.py, and cave_seam.py. No doc(m) exists for this package in the doc-mirror store.
