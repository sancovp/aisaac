THE NEW VIEW

The realization was architectural: each organ should be its own CronAutomation firing on the Heart tick, not one script doing everything. The framework-first gate ‚Äî no socials until the sibling journey_core.json has both deep_dive_url and plugin_url ‚Äî was the admissibility guard that prevented packing a blog whose CTAs point nowhere. The canonical-core enforcement ‚Äî the module overwrites whatever the AI agent wrote with the authoritative JSON before rendering ‚Äî was the fix for a run where the agent retyped the core from memory and introduced nine field corruptions including a broken URL. The never-raises contract ‚Äî every fire_* function catches exceptions and writes them to the CartON node ‚Äî was the contract that lets the Heart tick keep running even when one node fails. These three constraints are what makes the pipeline monotone: each tick only accretes properties, converging when nothing is pending, never retracting.

‚Äî testing ‚Üí

THE RIGHT WAY

Three CronAutomations, each a distinct organ, firing in sequence every night. Site publish (02:37): renders blog markdown to HTML, inserts an idempotent index card, commits and pushes to GitHub, flips site_published on the node, fires a Discord announce. Socials pack (03:07): reads live blog posts, dispatches a heaven/MiniMax agent to fill journey_core.json and journey_blog.json, enforces the canonical core on disk, renders the social suite deterministically ‚Äî twitter thread, LinkedIn post, Discord copy, image prompts ‚Äî flips socials_packed. Images (03:37): runs a claude -p excalidraw agent on the image prompts, verifies PNGs exist, flips images_rendered. The framework-first gate holds any not-yet-linked post and retries the next night. The Heart tick never crashes. Content flows from done node to live URL to reader screen without a human in the loop.
