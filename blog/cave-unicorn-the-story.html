The Distribution Pipeline

You can spend a year perfecting the content organ, but if there is no last-mile distribution layer, nobody ever reads it — and you have built a ghost that nobody can see.

OVERVIEW

The pain: You spend months perfecting a content-producing pipeline — blogs, deep dives, social posts — and then the output just sits in a directory. Every night the organ fires and nothing reaches a reader. Getting the content live still means manual HTML conversion, hand-editing post skeletons, git commits, and pushes. Every single time. You have built a pipeline that produces content but never delivers it.

The dream: The journey from a journal entry to a post a stranger can find is automated. Content arrives at a reader's screen every night without a human in the loop. The system has an audience it can grow with.

My solution: Built cave-unicorn — the machine that takes a finished piece of writing and delivers it while I sleep: the post goes live, the announcement fires, the social copy and the images appear, every night, hands-free. We solved the last mile — the first piece flowed from finished draft to live URL in one night.

THE DEBATE

The last-mile gap kept getting deferred. The content organ was more important. Other things were on fire. By the morning of July 10, 2026, I had landed 25 prior tasks and the distribution workstream was still a paragraph in a journal entry. The hesitation had two sources: it felt like a simple problem — surely I could just push a file — and it was not clear which website to wire or where the Discord library actually lived. The gap kept expanding because it kept looking smaller than it was.

THE TRIALS

The grounding work took the first half hour. UNICORN existed in the cave architecture doc as a business-game name and a score row — there was no module design. I traced the GitHub Pages site to sancovp/aisaac and found it already had a blog directory with 20 published posts and a blog-style.css. The Discord library had three candidates on disk — cave-discord, cave-discord-fork, integration/discord-mcp — and the authoritative version was resolved by tracing the source-inversion commit of July 7, 2026, which had rescued a stale laptop copy while declaring the monorepo copy canonical. The monorepo copy was right. Then I built the four core files — site.py, publish.py, cave_seam.py, cli.py — in about 40 Write and Edit calls and installed cave-unicorn 0.1.0 with 24 passing tests. The first E2E test immediately revealed a GitHub Pages build failure caused by Jekyll processing unfinished checkpoint markdown files with brace syntax — the fix was adding .nojekyll to the repo root. A dangling deep-dive link in the first published post was cured by a --hook flag and link_map republish. The Discord hookup used cave_discord.client.DiscordMCPClient to post to the coglog channel via the new library, wired as cave_unicorn.announce firing after every pushed publish. By 05:55 the first two blog posts were live on the site and the Discord announcement had fired. The socials leg required finding the socials-pack skill, grounding its image prompts against the blog organ output, and fixing a date-hallucination bug where the agent wrote 2025-07-09 instead of the real 2026-07-10. A real heaven 0.1.35 regression was discovered during the automation work: standalone hook scripts were calling sys.exit(0) at module level, and that BaseException escaped both callers except-Exception handlers and silently killed every automation dispatch from the home directory — fixed in heaven 0.1.36, verified on the exact previously-dying surface. The three automations — unicorn_site_publish (02:37), unicorn_socials_pack (03:07), and unicorn_socials_images (03:37) — were emitted to the hot-reload directory and the full pipeline proved hands-free by 06:25 on July 10: one piece of content, node to live URL to Discord to social posts to PNGs, every stage automation code.

THE NEW VIEW

The realization was architectural: each organ should be its own CronAutomation firing on the Heart tick, not one script doing everything. The framework-first gate — no socials until the sibling journey_core.json has both deep_dive_url and plugin_url — was the admissibility guard that prevented packing a blog whose CTAs point nowhere. The canonical-core enforcement — the module overwrites whatever the AI agent wrote with the authoritative JSON before rendering — was the fix for a run where the agent retyped the core from memory and introduced nine field corruptions including a broken URL. The never-raises contract — every fire_* function catches exceptions and writes them to the CartON node — was the contract that lets the Heart tick keep running even when one node fails. These three constraints are what makes the pipeline monotone: each tick only accretes properties, converging when nothing is pending, never retracting.

THE RIGHT WAY

Three CronAutomations, each a distinct organ, firing in sequence every night. Site publish (02:37): renders blog markdown to HTML, inserts an idempotent index card, commits and pushes to GitHub, flips site_published on the node, fires a Discord announce. Socials pack (03:07): reads live blog posts, dispatches a heaven/MiniMax agent to fill journey_core.json and journey_blog.json, enforces the canonical core on disk, renders the social suite deterministically — twitter thread, LinkedIn post, Discord copy, image prompts — flips socials_packed. Images (03:37): runs a claude -p excalidraw agent on the image prompts, verifies PNGs exist, flips images_rendered. The framework-first gate holds any not-yet-linked post and retries the next night. The Heart tick never crashes. Content flows from done node to live URL to reader screen without a human in the loop.

THE BOON

The bottleneck was never the content — it was the last mile. Now the pipeline runs every night and I do not touch it. A done Blog_Request node auto-publishes, auto-announces, auto-packs, and auto-renders images. When a new framework blog is ready, one cave-unicorn publish command wires it to the live site and the social pipeline activates the next night. The system has an audience it can grow with — and an audience changes what you build, because now you know someone is watching.

The problem it solves: a system that produces content but has no path to readers. The specific thing it solves: the last-mile gap between a done Blog_Request node and a live URL.
