SkillTree · A free pattern

Your agent isn't dumb. Its geometry is disorganized.

The felt problem

A flat pile of skills is a context bomb.

When every skill sits in one flat list, the agent has to hold the whole list in its head to pick one. Ten skills is fine. Fifty is a wall of names it half-remembers. The pile lands in context all at once — and the more you add, the more the model has to read before it can even start.

You've seen the failure mode: context overload, then the wrong selection, then a confident execution of the wrong thing. The agent isn't confused about your goal — it's confused about which tool answers it, because you handed it a heap and asked it to find the needle every single turn.

The pattern

Give the toolbox a coordinate system.

SkillTree is a tree of skill-directories instead of a flat pile. Skills get coordinate-addressed — a place in the tree, not just a name in a list — and the agent reads them by progressive disclosure: it loads the index, sees the branches, and drills into exactly one. No spoilers, no wall.

Progressive disclosure

Load the index first. See branches, not fifty full skill bodies. Drill one branch when the task actually calls for it. The agent reads what it needs and nothing else.

Coordinate-addressed

Every skill has a location in the tree, not just a name. The agent navigates to a capability the way you navigate a filesystem — geometry it can reason about instead of a list it has to memorize.

Scoped search

Search inside a branch instead of across the whole pile. The agent narrows to a subtree first, so lookups stay small even as the total skill count grows.

O(depth), not O(#tools)

This is an architectural claim, not a benchmark: to reach a skill the agent traverses the depth of the tree, not the width of the whole toolbox. Add a hundred more skills to other branches and the path to any one skill doesn't get longer.
